# Hoist CSS — promote page styles into the shared stylesheets

The user keeps shared styles in three source files and page-specific styles in
each page's inline `<style>` block. This skill moves a **named** style out of a
page and into the right shared file, generalising it as it goes.

**Only act on the style the user explicitly names.** Never sweep a page and
hoist things on your own initiative. When unsure, **ask** (see §5).

## 1. The files

Edit the **source** files only — never `ui/dist/` (it is generated by `make`).

| File | Holds | Selector shape |
|---|---|---|
| `ui/root.css` | **Theme-independent** shared styles | plain selectors (`.warning.block`, `.art-toc`, `#sun-panel`) |
| `ui/almanac.css` | **Almanac-only** theme styles | `.almanac xyz { … }` |
| `ui/cosmos.css` | **Cosmos-only** theme styles | `.cosmos xyz { … }` |
| page `<style>` (e.g. `ui/articles/pic.html`) | styles used on **that page only** | plain selectors |

Pages (`ui/home.html`, `ui/contact.html`, `ui/unlog.html`,
`ui/articles/*.html`) all `<link rel="stylesheet" href="index.css" />` and carry
an inline `<style>` block. `index.css` is built by `make` from
`root.css` + `almanac.css` + `cosmos.css` (the compose step injects the two theme
files at the `/* {{almanac-css}} */` and `/* {{cosmos-css}} */` placeholders in
`root.css`). So anything landing in a shared file is available to every page
after a rebuild.

## 2. Where does the named style go?

Decide with this order:

1. **Used on only one page → leave it.** If the class/selector appears in just
   one page and there is no reason to share it, it stays in that page's
   `<style>`. Tell the user; don't move it.

2. **Theme-independent → `ui/root.css`.** The rule is the same regardless of
   theme. **This includes rules that reference theme CSS variables** (`--ink`,
   `--paper`, `--accent`, `--rule`, `--serif`, …): if only the *variable values*
   differ per theme but the rule body is identical, it is theme-independent — put
   the single rule in `root.css` and let the variables resolve per theme. A rule
   is theme-independent when you would write the exact same declaration block for
   both almanac and cosmos.

3. **Theme-dependent → split into `ui/almanac.css` + `ui/cosmos.css`.** The rule
   itself differs between themes (different property values, layout, or design —
   not just a variable that already resolves per theme). Write the almanac
   version as `.almanac <selector> { … }` in `almanac.css` and the cosmos version
   as `.cosmos <selector> { … }` in `cosmos.css`. Use **descendant** selectors
   (`.almanac .eyebrow`), never flat prefixed names (`.alm-eyebrow`) — see the
   project's theme-scoping convention.

If a style is *mostly* shared with a few theme-specific bits, hoist the shared
declarations to `root.css` and push only the differing declarations into the two
theme files as overrides. Don't duplicate the shared part.

## 3. Always check other pages first

**Before generalising, grep for every usage of the selector/class across all
pages and stylesheets** — the point of hoisting is to serve more than one page,
and other pages may already have their own copy or a near-duplicate.

```
grep -rn 'warning\.block\|class="warning' ui --include=*.html --include=*.css
```

Check for:
- **Other pages using the same class** → they'll pick up the hoisted rule; make
  the hoisted version cover their needs, and **remove their now-redundant inline
  copies**.
- **Near-duplicates with drift** (same class, slightly different values) → this is
  a generalisation decision. Reconcile into one rule if they should match; if the
  differences are intentional/page-specific, **ask** which wins.
- **An existing rule already in a shared file** → extend/reuse it instead of
  adding a second rule (see §4, reuse).

## 4. Placement & conventions (from CLAUDE.md)

- **Reuse first.** Before adding a rule, check the destination file already
  provides it (a class, a variable, an existing rule). Use what's there.
- **Order within `root.css`:** Variables → General (shared base) → theme
  placeholders → shared component sections (Article nav/hero/meta/TOC/body/CTA,
  etc.) → Responsive `@media`. Drop the hoisted rule into the section it belongs
  to, near related rules, under a matching `/* ── … ── */` comment banner
  (add one if the group is new). Keep `@media` rules in the Responsive block at
  the bottom.
- **Order within `almanac.css` / `cosmos.css`:** group with related theme rules;
  these files hold only what differs from the general styles.
- **Spacing uses `--ms*`** modular-scale variables (`--ms-7` … `--ms7`), never
  hard-coded rem/px. If no step fits, ask.
- **Font sizes use `--ts*`** variables (`--ts15`, `--ts36`), never hard-coded.
- If a needed base style/variable doesn't exist yet, **ask** whether to add it to
  `root.css` or keep it inline — don't decide alone.

## 5. Workflow

1. **Locate** the named style in its origin page's `<style>` block. Read the full
   rule(s) — including any `:hover`, `:focus`, child, and `@media` variants of
   the same component.
2. **Grep other pages** for the selector/class (§3). Note every other usage and
   any drift.
3. **Classify** theme-independent vs theme-dependent vs single-page (§2).
4. **Confirm conventions:** `--ms*`/`--ts*` usage, reuse of existing rules,
   descendant theme selectors. Fix violations while hoisting (e.g. convert a
   hard-coded `16px` gap to the matching `--ms*`), but flag anything non-obvious.
5. **Move it:**
   - Add the generalised rule to the destination file in the correct section.
   - **Remove** the rule from the origin page's `<style>` — and from any other
     pages whose inline copy the hoisted rule now supersedes.
   - Preserve `:hover`/`:focus`/responsive companions; move them too.
6. **Rebuild** so `ui/dist/` reflects the change: run `make` (the `dev`/`TRX`
   target). Never hand-edit `ui/dist/`.
7. **Report** what moved where, which pages were affected, and any drift you
   reconciled or left alone.

## 6. Ask when unsure

The user explicitly wants to be asked rather than guessed at. Ask when:

- It's ambiguous whether a style is theme-independent or genuinely differs per
  theme.
- Near-duplicate rules on different pages have drifted and it's unclear which
  values are canonical.
- Hoisting would need a new base rule or a new `--ms*`/`--ts*`/colour variable.
- No `--ms*`/`--ts*` step matches a value you'd otherwise hard-code.
- A style *looks* shareable but is only used on one page — confirm before moving
  vs. leaving it.
